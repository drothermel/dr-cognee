What worked — keep doing 6 keepers
Keeper 01 · highest-value agent of the run

The code-grounded skeptic

The feasibility critic read sources.py/ingest.py/docs_pull.py and the four real workspaces on disk, and found things no amount of careful design would have surfaced. Four design-breaking facts, all sitting in the repo the whole time:

Torn reads guaranteed: SourceStore.update is a non-atomic truncate-then-write
71/88 in polars: docs sources are born scraped and never distilled
~5 vs 263 id citations: tool-graph reports cite [synthesis/*.md], not source ids
Hashes only: the ingest manifest carries hashes, nothing else
Keeper 02

Live verification beat documentation reading

R8 probed the hosted tenant's openapi.json and made real GET calls instead of trusting docs. It settled the project's biggest unknown — graph export: yes, verified, 3k nodes in one call — and killed a false assumption (the INSIGHTS search type no longer exists). Two hours of read-only probing changed the entire selection calculus.

Keeper 03

Brainstorming candidates before research landed

02-usecase-candidates.md was written while agents were still running, so the use-case space came from the problem structure (moments × data shapes), not from whatever the agents happened to find. Research then validated the bundles instead of dictating them — and where it changed things (D5 demoted from fallback to preview layer), the delta was visible.

Keeper 04

Critique before pitch-building

Two threatening and ~10 material findings got fixed in the designs before a single line of pitch HTML existed. The pitches now carry honest budgets (v1 estimates roughly doubled) and "what the critics said" sections. Reversed order would have shipped confident wrongness.

Keeper 05

Docs-as-you-go made delegation cheap

Because every phase wrote its artifact to docs/viz/, the five HTML builders needed zero conversation context — just file paths. The workspace, not the chat, was the source of truth. (This is, pleasingly, the same lesson drc itself embodies.)

Keeper 06

Convergent evidence from overlapping agents

R1 and R7 independently arrived at the same architecture (local FastAPI, file-watch → SSE, zero storage); R1 and R6 independently arrived at master-detail + counts-not-percents. The overlap I'd half-considered trimming became the strongest signal in synthesis: independent agreement is worth its token cost.

The structure mostly held. The left lane is what earned its place; the right lane is what I'd run differently, each failure paired with the fix that was either adopted mid-run or is adopted now, in this document. The honest headline is that the best and worst moments of the run share a root cause: the real data on disk. The feasibility critic's four design-breaking facts came from reading the actual workspaces; my worst mistake was designing against models.py instead of profiling those same workspaces in hour one.

Clean, correct, and misleading — that's what a data inventory read off the models is. The distribution facts that mattered were on disk the whole time. — on failure 2, the schema-vs-data trap

The second thread is ordering. Everything good about the run was an ordering decision: candidates before research, critique before pitches, verification before trust. Everything bad was an ordering omission: profiling after design, spikes after critique, doc updates after "later." The process amendments at the bottom of this page are all, in one way or another, "move this step earlier."

The third thread is economics. Wave-1 agents relayed full documents through my context and I paid for every research doc twice; the HTML wave fixed it by having agents write their own files and return five-line summaries. The case studies were maybe 40% restatement of wave-1 findings — real value in the other 60% (screen anatomies, walkthroughs, CS2's verification of Cognee's actual visualize template source), but a wave that should have been folded into the first.

Where this leaves the process

If I strip the run to its load-bearing moves, I get: a skeptic with repo access, a live probe of the one API that mattered, my own hypothesis written before the evidence arrived, and a workspace of docs any agent could pick up cold. Those four survive any re-design of the process. The rest — wave counts, agent counts, the exact phase boundaries — is negotiable, and the alternatives strip below negotiates it.

And then there is the question the critique forced, which no amount of process tuning answers — rendered in full at the bottom of this page, because it is the actual thesis of this report: research produced conviction about shape; only usage produces conviction about value.

What went wrong → the fix 5 failures, 5 fixes
Failure 01 · stale ground truth

I let a ground-truth doc go stale, and it poisoned a critique

R8's findings invalidated the central caveat of 01-data-inventory.md ("graph export not available — biggest feasibility unknown"), and I didn't circle back to update it. The UX critic then built its most severe atlas finding on the stale sentence.

Fix →

Adopted mid-run, worth institutionalizing: when new evidence invalidates a foundational doc, the update happens in the same turn as saving the evidence.

Failure 02 · schemas over data

I designed against the models, not the data

My data inventory came from reading models.py and workspace.py — clean, correct, and misleading. The real workspaces (which existed on disk the whole time!) held the distribution facts that mattered: most sources in tool workspaces never pass through distill; real reports vary wildly in citation density. The feasibility critic did the workspace profiling I should have done in hour one.

Fix →

The data inventory phase should include profiling actual instances, not just schemas — a 20-minute script over research/.

Failure 03 · paying twice

Token-expensive relay of research docs

Wave-1 agents returned full documents in their final messages, which I then re-emitted verbatim into files — paying for every research doc twice. The HTML wave fixed this (agents Write their own files, return a 5-line summary).

Fix →

Any agent producing a >500-word artifact should write it to disk itself; the coordinator gets a summary and spot-checks.

Failure 04 · late de-risking

Spike-thinking arrived late

The critique pass produced the consolidated "1 day of spikes gates 3 weeks of builds" list — watcher torture test, source_content_hash check, citation-coverage script, rapidfuzz hit-rate. All four are runnable against data already on disk, and all four should have been in the design template.

Fix →

Amend the template itself: "§5 must name the riskiest bet and the cheapest spike that retires it." The template has been amended in my head; it should be amended in the doc for next time.

Failure 05 · redundant wave

Case studies were half-redundant

Wave-2's three case studies produced real value — the concrete screen anatomies and fictional walkthroughs fed the pitch mockups directly, and CS2 uniquely verified Cognee's visualize template source — but maybe 40% of their content restated wave-1 findings.

Fix →

Fold "include one concrete walkthrough + exact screen anatomy of the single best exemplar" into the wave-1 prompts, and reserve wave-2 for verification-style deep dives only.

If I ran this again — alternative approaches

5 counterfactuals, each with a verdict
← scroll → · five ways this run could have been structured, and whether I'd take each one
A

Data-first

Would adopt

Phase 0 becomes: profile the real workspaces with scripts — status distributions, citation formats, file sizes, manifest shapes — and probe the live API. Then write the framework.

Why adopt: half the critique findings would have been design inputs instead of design corrections. This is failure 02 turned into a starting move.
B

Prototype-first

Consider — mission only

For a personal tool with live data sitting on disk, a half-day throwaway cockpit spike might teach more than two research agents. The rubric and market scan earn their cost for the portfolio decision, but the watcher plumbing questions were always going to be settled by code, not prose.

Verdict: a hybrid — research the selection, prototype the riskiest slice during the design phase — is probably optimal.
C

Structured outputs over free markdown

Would adopt partially

Findings-style agents — the critics, the API surveyor — could return schema-validated objects (finding/severity/evidence/fix), making the decision pass mechanical instead of editorial.

The partial: survey-style agents should stay free-form — their nuance is the value.
D

Fewer, deeper agents

Would not adopt

Collapsing 8 wave-1 topics into 4 broader ones would halve coordination cost — but it would lose the independence that made convergence meaningful (keeper 06), and single agents skim when given two fields to cover.

Why not: the R1/R7 and R1/R6 independent agreements were the strongest signal in synthesis. Merging the topics deletes the signal to save the tokens.
E

Interactive checkpoints

Situational

This run was fully autonomous by request. With the user present, two checkpoints would have earned their interruption cost: after selection (before three design deep-dives) and after critique (the accepted-changes list rewrote real scope).

Everything else was better off autonomous.
The uncomfortable question · the thesis of this report

Build one week of it, then look at which tab stays open.

Three pitches totaling ~3 weeks of build effort now exist for a personal tool whose owner has a finite hobby budget. The critique's sharpest cross-cutting point — build the shared-chassis v0 week and let usage decide which v1 exists — is really a statement about this whole exercise:

the research produced conviction about shape, but only usage produces conviction about value.

The honest summary of three weeks of design work is: build one week of it, then look at which tab stays open.
