DRC VIZ FAMILY · REPORT 2 OF 5 · USE CASE: GRAPH & KNOWLEDGE EXPLORER

drc atlas

Search first, expand on demand, evidence one click away.

A local, search-first explorer over per-topic Cognee knowledge graphs. Text answers stay primary — drc query is unchanged — the atlas is the verification, orientation, and serendipity layer around them: what does this graph know, what is the evidence behind any node, what is thin or missing, and what did the last cognify change. Never a hairball; the canvas holds only what you asked for.

Act 01

The moment this exists for

cognify just finished — now what does it actually know?

Q?The five questions, ranked

Every design element traces back to one of these. Anything that serves none of them was cut.

  1. What does this graph know about X?
    omnibox → ego scene + evidence
  2. What's the evidence behind this node or edge?
    provenance breadcrumb
  3. What are the major themes — and what's thin?
    community overview + gap cards
  4. How do X and Y connect?
    path mode, straight strand
  5. What did the last cognify change?
    scene-scoped diff
[Atlas §1] [R3: search-first]

The anti-goal: the hairball

3,033 nodes · 13,189 edges · rendered all at once = noise

Global force-layout renders fail past ~200 nodes. Obsidian's graph view is the canonical cautionary tale: the most famous personal knowledge graph in the world, universally praised as beautiful — and near-universally admitted to be useless for actual work. A screenshot generator, not a thinking tool. drc's mid-size dataset is 15× past the legibility cliff.

So the atlas takes Bloom's rule as law: the canvas holds only what was searched or expanded. Overview is meta-nodes, never raw nodes. A beautiful global render is an explicit non-goal, as is 3D.

[R3: hairball anti-goal] [CS2: Bloom scene model] [Atlas P1]

Why the built-in visualize isn't enough

Cognee ships GET /visualize — a self-contained d3.v7 force page (verified live, 384 KB). CS2 read its source. The complete interaction list:

pan / zoom  (d3.zoom on the whole container)
drag a node  (pins during drag, releases after)
hover tooltip = JSON.stringify(d)  (raw props blob)
search · filter · click-to-inspect · type toggles · legend
expand-from-selection · clusters · gaps · saved views
provenance navigation of any kind

At drc scale it renders every node with a 5px label and every one of 13k edge labels in SVG per tick — a hairball and a frame-rate disaster. What it does prove: cognify ran, types attach, the graph exists. The atlas inherits its one good idea — the type-color convention — and replaces everything else.

Act 02

How it works

three ideas: scenes, two data layers, a provenance chain with named confidence

01The scene model — never render what wasn't asked for

Borrowed whole from Neo4j Bloom, the reference design for search-first exploration: a scene contains only what you searched for or expanded to. Landing state is an omnibox over an overview of meta-nodes (EntityTypes and Louvain communities sized by member count, colored by dominant node set) — available from /schema/inventory before a full export even exists. The raw graph is never the first paint.

Search
typeahead over labels + descriptions; Enter materializes the hit
Scene
hit + ranked 1-hop context, DOI-capped at ~30–50 nodes, focus pinned center, radial layout
Expand
right-click a frontier node → all / by edge label / entities-only (skip plumbing)
Inspect
click → description, weights, edges by label, the Sources breadcrumb
Share
scene serialized to the URL hash — every view is a pasteable artifact

Scenes are queries, not id lists [05: accepted] — the URL hash stores {search, expansions:[(label,type,edge)], lenses, camera} and re-resolves on load. Cognee node ids are not stable across cognify runs; a saved scene made of ids would rot in a week. A saved scene made of queries survives re-cognify, and a missing label degrades to a note, never a broken link.

DocumentChunk / TextSummary / TextDocument nodes — ~40% of the graph — are provenance plumbing, not knowledge. They are hidden from the canvas by default and surfaced as the inspector's Sources breadcrumb instead. [Atlas P2] [CS2 mapping]

Layer 0 · local co-occurrence — the pre-cognify preview

Works before Cognee ever runs

Built from distilled/*.json already on disk: entities that co-occur in the same distilled record get an edge. Free, instant, offline — and it is the atlas's empty state: same scene UI, same omnibox, a banner reading "local preview — cognify to get the full graph." The explorer is useful on day one of a topic, not just after a paid cognify. [05: D5 accepted]

Layer 1 · Cognee snapshot — the enhanced mode

One GET, cached as a local snapshot

GET /datasets/{id}/graph → viz/graph-<n>.json, gzipped and property-stripped, written by the "refresh ⟳" button or drc ingest --snapshot. FA2 layout runs client-side in a web worker on first load; positions persist back into the snapshot file, so spatial memory survives reloads — overview and diff both consume the same stored positions. [R8: verified live] [R2: positions persisted] [05: FA2 resolved]

Search is the only thing that ever costs credits — and only on explicit clicks in the workbench, never on polling. [Atlas P5]

02The provenance chain — every claim walks home

Question #2 is "what's the evidence behind this node?" — and it is the feature the whole pitch stands on. Cognee's chain ends at TextDocument; drc's workspace begins at ingested.json. The atlas joins them server-side in one shared provenance.py chassis service (the dossier consumes the same service, so the two surfaces can never disagree). [05: one provenance service]

Act 05

Eight decisions, with receipts

every one traceable to research, a case study, or an accepted critique
D·01

Search-first, never the hairball

The empty scene + omnibox is the front door; the full graph is never rendered. Global renders fail past ~200 nodes and this dataset is 3,033. Bloom's scene model is adopted as law, and Bloom's cold-start weakness is patched by decision D·03.

[R3: search-first] [CS2: Bloom] [Atlas P1]
D·02

The ego scene is the workhorse

Q1 ("what does it know about X?") is the top-ranked question, so its answer shape — hit + ranked 1-hop, DOI-capped at 30–50 nodes, focus pinned, radial — is the screen you live in. Expansion is per-edge-label, reversible, recorded.

[Atlas §3] [CS2: expand-from-selection]
D·03

Communities as the overview

InfraNodus's insight: the first paint should already be an answer — "your graph is about these six things." Louvain over the Entity–Entity projection, meta-nodes sized by count; betweenness picks the brokers; gap detection turns topology into the next harvest query.

[CS2: InfraNodus] [Atlas §4]
D·04

Diff, not animation

R4 surveyed dynamic-graph viz and the verdict was unambiguous: animated transitions destroy comprehension; small-multiple / union diffs win. Atlas ships a static, scene-scoped union overlay with positions synced to the newer snapshot and a ranked "most rewired" entry list.

[R4: diff>animation] [05: scene-scoped]
D·05

Lenses derived from node sets

drc already tags every push — docs, distilled, content, synthesis, log, report — and the export carries the tags on every node. Lens chips are generated from the data, never hardcoded (the critique caught a 4-of-6 hardcode). Report lens = "what survived into the report."

[R8: belongs_to_set verified] [05: accepted] [CS2: perspectives]
D·06

Scenes are queries — they survive re-cognify

Cognee node ids are not stable across cognify runs. Scenes serialized as id lists would rot; scenes serialized as {search, expansion steps, lenses, camera} re-resolve against any snapshot. A missing label degrades to a note, never a broken link. Bookmarks stay honest for the life of the topic.

[05: accepted] [Atlas §6]
D·07

Searches are lazy and priced

Hosted credits are real money and the tenant returns 402 when exhausted. So: snapshots power everything free; workbench tabs fire one billed search per explicit click (never three types at once); a per-session cost counter sits in the drawer; 402 maps to CogneeCreditsError with a visible banner.

[Atlas P5] [05: triple-billing fixed]
D·08

sigma.js + graphology, MIT

WebGL rendering that holds 3k/13k without breaking a sweat, a real graph model with Louvain + FA2 + betweenness in the same MIT family, from CDN with zero Python deps added. FA2 runs in a web worker once per snapshot; positions persist — spatial memory is a feature, not luck.

[R2: stack survey] [Atlas P4] [05: FA2 resolved]

Act 07

What the critics said — and what changed

every finding got a verdict; the two hardest are here in full

Objection 1 — the demand problem

"Graph explorers demo beautifully, then go unopened. Graph browsing is the weakest-demand job of the three." — the skeptic, feasibility critique · verdict: MODIFY (conceded in the pitch, not designed away)

The concession is structural, not rhetorical:

  • ▸Build order: 3rd of 3. Mission and dossier ship first; atlas waits its turn and must justify it.
  • ▸Named survival criteria. The provenance breadcrumb and the gap cards are the two features that must earn reopening. If they don't get used, the atlas doesn't graduate past v0 — the criteria are written into the pitch, not implied.
  • ▸The spike gates the investment. Two hours of hash-checking before any of the 9 days is spent.
  • ▸And the demand story is deliberately not "browsing": it's five specific questions, two of which (evidence, gaps) no text interface answers well.
[05: skeptic MODIFY] [Atlas §6: honest budget]

Objection 2 — "the join is built on a hashes-only manifest"

"The provenance join assumes ingested.json can be matched to TextDocuments — but the manifest records only digests. Rated: threatening." — feasibility critique · verdict: ACCEPT, plan reordered

Accepted in full, and the fix is ordered so cheap certainty precedes expensive engineering:

  1. 2-hour spike: the export already carries source_content_hash on entity properties (verified live). If it matches the manifest digests, the "threatening" join is a dict lookup and the objection dissolves.
  2. Else, v1 joins stable keys by re-hash — docs / distilled / content files are immutable on disk, so re-hashing reproduces the digest. Synthesis / log / report artifacts get rewritten between pushes → labeled "ingested artifact (revision unknown)"; orphan TextDocuments from add-only re-pushes labeled "superseded revision" and excluded from counts by default, toggleable.
  3. Pipeline upgrade: record the Cognee data-id in ingested.json at push time — one line at push, exact provenance forever after.
