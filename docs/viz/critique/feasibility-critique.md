Reviewed against the actual code on `07-07-tool-graph` and the real workspaces under `research/` (agent-graph-knowledge-base: 229 sources; polars: 93; pydantic: 99; pydantic-ai: 184). The good news first: the Cognee API research doc is genuinely load-bearing and checks out against `cognee_client.py`; `found_via` string formats claimed in the inventory match `docs_pull.py:82,119` and `firecrawl_ops.py:52` exactly; real source ids are 12-hex as `sources.py:11` (`SOURCE_ID_LENGTH = 12`) implies. Now the attack.

## drc mission (Run Cockpit)

**[threatening] Torn reads of sources.jsonl during watch.** The design's P2 fix ("full reparse on change") assumes the file is always readable. It isn't: `SourceStore.update()` at `src/dr_cognee/sources.py:61-67` does `self.path.write_text("\n".join(lines) + "\n")` — a truncate-then-write with no tmp-file+rename and no locking. During a distill loop, `update()` fires once per record; watchfiles will regularly wake the watcher mid-write and `SourceRecord.model_validate_json` will throw on a half-written line (or the watcher silently sees an empty/truncated file and emits phantom "source removed" events). — Fix: watcher must treat parse failure/shrunken file as "retry after debounce," and separately, add atomic-rename to `SourceStore.update` (a 3-line change that also removes the crash-loses-everything failure mode).

**[material] Watcher writes into the directory it watches.** `viz/events.jsonl` lives inside `<workspace>/` per the design ("appends to `<workspace>/viz/events.jsonl`"), which the same `awatch` observes — classic self-triggering feedback loop, and `log_appended` events will fire for the watcher's own writes. — Fix: exclude `viz/` from the watch filter explicitly (one `watch_filter`), and say so in the design.

**[material] The four-stage funnel/grid model is wrong for tool-graph workspaces — i.e., most current workspaces.** Docs sources are *born* at `SCRAPED` (`docs_pull.py:48`: `status=SourceStatus.SCRAPED`) — they never pass through `FOUND` — and they skip `DISTILLED` entirely, jumping to `INGESTED` in bulk (`ingest.py:184-187`). In the polars workspace, 71 of 88 content-bearing sources have no distilled record. The mockup's `found 46 → scraped 41 → distilled 29 → ingested` monotone funnel and the F/S/D/I grid columns will show a wall of "stuck at scraped" for the run's dominant category, defeating question 2 ("what broke?"). — Fix: per-category stage semantics (docs lane: pulled → ingested), and render the funnel as counts-by-state (which the design already prefers), not a pipeline.

**[material] "Ingested" status is a lie relative to what the graph contains.** `ingest_workspace` pushes texts and updates the manifest *first*, then flips statuses only at the end — and deliberately leaves them un-flipped on a credits error (`ingest.py:170-174`: "pushes stand (manifest already updated); statuses stay un-flipped"). So the grid's I column can show "not ingested" for sources whose text is already in Cognee, and it flips all-at-once, not per-source. The Now panel's per-source ingest progress cannot exist from this data. — Fix: derive "pushed" from `ingested.json` keys (`distilled:<id>` / `docs:<id>` / `content:<id>`), and show status-vs-manifest disagreement explicitly instead of trusting `status`.

**[minor] Stall inference false-positives on distill.** A single LLM distill call can exceed 90s with zero file activity; the amber warning will cry wolf every long page. Cognify is handled ("possibly waiting on cognify") but distill isn't. — Fix: scale the threshold by phase, or by last-event kind.

**[minor] Dash needs Cognee env or it dies at import-time of the status widget.** `CogneeClient.__init__` does `os.environ["COGNEE_BASE_URL"]` (`cognee_client.py:31-32`) — KeyError, not graceful. Also `dataset_status` needs a dataset UUID, which today is only obtainable via `POST /datasets/` (a create-or-get write). — Fix: lazy-construct the client, degrade the widget to "not configured," and cache the dataset id from `topic.yaml`'s `dataset_name` once.

**Effort check.** v0 (1 day) is achievable *if* it's truly just funnel+grid+inspector with naive polling. v1 "2-3 days" is fantasy: Watch/Triage modal keyboarding with j/k, six verbs, auto-advance, an editable undo strip, since-you-left delta accounting via localStorage, entity pulse aggregation, and facet-rail filtering is a week of UI-state work at honest pace. Call v1 5-6 days.

**Riskiest bet:** semantic event derivation from non-atomic whole-file rewrites. **Cheapest spike (half a day):** a 60-line script that runs `awatch` against `research/polars/` while replaying a scripted `SourceStore.update` loop, printing derived transitions and counting parse failures/phantom events.

## drc atlas (Graph Explorer)

**[threatening] The provenance join's central claim is false: "the manifest records exactly what text went in."** It records *hashes only*. `IngestManifest` is `RootModel[dict[str, str]]` — "Maps ingested artifact key -> sha256 of the text last pushed" (`ingest.py:136-138`). There is no stored text, no Cognee data-id, no title. To join TextDocument ↔ source id you must either (a) re-run `build_payloads()` against the *current* workspace and re-hash — which breaks for `synthesis:*`, `log`, and `report` keys the moment those files are edited after ingest (they always are; that's the whole workflow), or (b) fetch `GET /datasets/{id}/data` + `/raw` per document and hash server-side. Worse, because ingest is add-only (no update/delete anywhere in `ingest.py`), a re-pushed synthesis file leaves the *old* TextDocument in the graph with no manifest entry at all — permanently unjoinable, and silently double-counted in every node-set lens. — Fix: v1 join only the stable keys (`docs:*`, `distilled:*`, `content:*`) via re-hash, mark synthesis/log/report docs "ingested artifact (revision unknown)"; longer-term, record the Cognee data-id or a `pushed_at` snapshot of the text alongside the digest at push time (small `ingested.json` schema change).

**[material] Possible shortcut left unverified.** The research doc says entity properties include `source_content_hash` (`cognee-api-surface.md:38`). If that hash is sha256-of-pushed-text, the whole join collapses to a dict lookup against `ingested.json`. Nobody checked. — Fix: this is the spike (below); do it before writing any fuzzy-matching code.

**[material] Node-set lens list omits half the actual sets.** The design's lenses are "(docs)(distilled)(synthesis)(report)" (§3, §4 mockup) but `ingest.py:20-25` defines six: `distilled, content, docs, synthesis, log, report`. `content` is precisely the interesting one — depth-flagged high-relevance full sources — and `log` is noise you'd want to *exclude*, which you can't without a chip. — Fix: generate lens chips from the sets present in the export (`properties.belongs_to_set`), never hardcode.

**[material] Search costs and errors unhandled in the workbench path.** `CogneeClient.search` (`cognee_client.py:94-102`) passes no `verbose`/`nodeName`/`topK` (design admits this) but also has no 402 handling — only `cognify` maps PAYMENT_REQUIRED to `CogneeCreditsError` (`cognee_client.py:58-59`). A credits-exhausted tenant turns the query workbench into raw `httpx.HTTPStatusError` tracebacks mid-page. Also `REQUEST_TIMEOUT_S = 120` per call with search latencies of many seconds means the FastAPI worker must proxy these async or the SSE/mission routes on the same server stall. — Fix: extend 402 mapping to `search`, run Cognee proxying on async httpx with a per-route timeout and an error banner.

**[minor] Empty/pre-cognify state undesigned.** pydantic's workspace is exactly this today ("cognify pending tenant credits" per its report.md): a dataset with pushed texts but no graph, or no dataset at all. The atlas needs a defined landing state for "no snapshot exists and refresh returns nothing useful." — Fix: specify an empty state that shows the local entity co-occurrence fallback from `distilled/` (the inventory doc already names it).

**[minor] Snapshot size on tool graphs.** 5.9 MB verified for ~3k nodes on `default_dataset`; a tool workspace ingesting ~90 whole doc pages (some 93 KB each, `research/polars/content`) plus distills will push the single-call unpaginated export well past that. FA2+Louvain in a worker on 10-20k nodes is fine; parsing/holding several 20+ MB snapshot JSONs for diff mode in one tab is where it gets ugly. — Fix: store snapshots gzipped, strip `properties` you don't render, and diff server-side.

**Effort check.** v0 "a weekend" for scene-model materialize/expand/inspector/lenses is plausible with sigma.js. v1 "+2-3 days" is not: the provenance breadcrumb alone (the join problem above, plus the `/data`+`/raw` fetch-and-cache layer) is 2 days by itself if the hash shortcut fails. Budget v1 at +5 days.

**Riskiest bet:** the graph↔workspace identity join. **Cheapest spike (2 hours):** one script against the existing polars/agent-graph datasets — fetch `/datasets/{id}/graph` + `/datasets/{id}/data` + a few `/raw`, compare `source_content_hash` and sha256(raw) against `ingested.json` digests, count exact matches. That number decides the whole provenance design.

## drc dossier (Report & Evidence Studio)

**[threatening] Tool-graph reports barely cite source ids — the core read-loop is empty in tool mode.** Measured on the real artifacts: `agent-graph-knowledge-base/report.md` has 263 12-hex id citations (dossier works beautifully there), but `polars/report.md` has ~6 and `pydantic/report.md` ~5 — and both instead cite `[synthesis/overview.md]`-style pointers (5 each), a citation form no part of the design parses. The tool-graph skill only requires "source pointers" without mandating ids (`docs/skills/tool-graph/SKILL.md:67`); only the deep-research skill says "cite source ids/URLs inline" (`deep-graph-research/SKILL.md:83`). Margin notes, valence chips, and the audit table render as blank chrome on exactly the long-lived reference workspaces the design says matter most. — Fix: parse `[synthesis/<facet>.md]` citations as first-class (link to the capability index), and add the recommended one-line skill edit to tool-graph *now*, not as an optional extra.

**[material] Docs sources have no DistilledRecord — the evidence-card ladder's level (a) doesn't exist for them.** P1's resolution is "the cited source's DistilledRecord is the hover card," but `build_payloads` ingests docs whole, "no distillation" (`ingest.py:83`), and in polars 71 of 88 sources have nothing in `distilled/`. Any docs citation — the dominant category in tool workspaces — has no takeaways, no claims_of_use, no relevance, hence no hover card, no valence, no grade numerator. — Fix: define the docs-source card explicitly (title, URL, `found_via`, first-heading excerpt from `content/<id>.md`) and exclude docs from valence rather than defaulting them to SECONDHAND.

**[material] The VENDOR valence heuristic requires data that exists nowhere.** "VENDOR when S's URL domain matches *the subject entity's domain*" — entities are bare strings (`entities: list[str]`, `models.py:63`); nothing in the workspace maps an entity name to a domain. As specified, this branch is uncomputable. Note the agent already does this labeling better than the heuristic could: the real report states "Vendor-supplied claims are labeled [vendor]... labeled [practitioner]" (agent-graph report.md header). — Fix: parse the report's own `[vendor]`/`[practitioner]` tags when present, and reduce the heuristic to computable proxies (`category == docs`, or source domain == GitHub org of the cited entity's repo when a github source exists).

**[minor] URL linkification must normalize before matching.** `sources.jsonl` stores normalized URLs — tracking params stripped, trailing slash removed, lowercased host (`sources.py:14-24`) — while report prose will contain raw URLs. Exact string matching silently misses. — Fix: run report-extracted URLs through `normalize_url` before the join.

**[minor] Fuzzy-match cost on tool content.** rapidfuzz of takeaway text against 93 KB content files × a few hundred citations is fine once, but "built on page load (or file change)" plus the mission watcher invalidating on every log.md append means recomputation storms. — Fix: key the cache per (citation, content-file mtime), not per workspace change.

**Effort check.** v0 (1.5 days) is honest *for deep-research workspaces*. The v1.5 "tool-mode capability index" estimate hides the real work: it's not parsing — it's that tool mode needs the alternate citation form, the docs-source card, and the synthesis-pointer linking described above, i.e., a second evidence model. +2 days becomes +4.

**Riskiest bet:** that agent-written reports are machine-indexable at all across runs. **Cheapest spike (1 hour):** run the planned extraction (12-hex regex + normalized-URL join + `[synthesis/...]` pattern) over the four existing report.md files and report per-report citation coverage — the data is already on disk, and it would have caught the tool-mode hole before design sign-off.

**Overall verdict.** All three are buildable; none is buildable *as written*. Mission's fixes are cheap but must land in `SourceStore` (the one place the "pure additions, no pipeline changes" promise is already broken).
