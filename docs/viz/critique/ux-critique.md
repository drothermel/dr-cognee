# Critique: the three drc viz designs (UX/product lens)

> Adversarial critique agent, saved verbatim 2026-07-07. Decisions in `../05-critique-response.md`.

Reviewer stance: skeptical senior product designer; single-user personal tool; rubric from `00-framework.md` §7 (decision value, data availability, weekend build, zero maintenance drag, wide-canvas payoff, distinctness).

---

## 1. drc mission — the Run Cockpit

**[threatening] Triage verbs have no write destination — the flagship loop doesn't persist.** §3 "Triage loop" defines `x` dismiss, `z` snooze, `n` note as first-class verbs with undo, but §5 says client state is "only `lastSeen` in localStorage; everything else server-rendered," and the design explicitly avoids mutating `sources.jsonl`. So a dismissed depth flag reappears on the next page load and the queue counts never durably shrink. The between-iterations triage session — the design's highest-decision-value moment — silently loses its work unless every decision is funneled into the copy-paste steering note. — *Fix: define a `viz/triage.jsonl` (or fold dismiss/snooze into the v1.5 directives file, pulled forward to v1) as the persistence layer for verb outcomes.*

**[material] `Esc` is overloaded with a destructive and a navigational meaning.** §3 re-engagement loop: "Pressing `Esc` … marks-all-seen and stores the new last-seen timestamp." §3 state diagram: `Esc` also exits INSPECT back to WATCH. A user drilling into an error and pressing Esc to close the inspector will wipe all since-you-left badges — exactly the delta they came back to review, unrecoverably (no unmark is specified). — *Fix: reserve Esc for pane dismissal and make mark-all-seen an explicit click on the badge chip only.*

**[material] "Thin facets" doesn't fit the queue interaction model.** §3 lists `Thin facets (2)` as a queue tab alongside source-shaped queues, but the verbs (`r` retry-suggest, `s` skip-suggest, `x` dismiss flag) are per-source actions; none is meaningful on a facet, and the detail pane spec (§4 Inspector = DistilledRecord) has no facet rendering. The interaction grammar contradicts itself. — *Fix: move thin-facet warnings to the facet rail's ⚠ markers only (already specified there), with a single "add to steering note" action.*

**[minor] The "trust channel" is buried at the bottom of the canvas.** §1 ranks "is the agent working right now?" as question 1 and §4 calls the log narrative "the agent's own voice, the trust channel," yet it's a tail strip at the bottom of a 1600px-tall screen while the Now panel — an *inference* from file events — gets the prime top-left slot. Ranking and layout disagree. — *Fix: pair the Now panel with the latest log.md line inline in the top strip.*

**[minor] The watcher watches its own output.** §5 appends `viz/events.jsonl` inside the watched workspace, so every event write is itself a file-change event; unmentioned exclusion logic, and events.jsonl grows unboundedly across a long-lived workspace. — *Fix: exclude `viz/` from the watch set and state a rotation/size policy.*

**Strongest skeptic objection:** the cockpit is a read-only mirror of a process whose actual control surface is the agent chat sitting in the next window. The chat already streams what the agent is doing (question 1), shows errors (question 2), and is where steering must land anyway — v1's answer to steering is composing markdown to paste *back into that chat*. After week one, the user plausibly just reads the transcript and asks the agent "any errors? what's thin?" The durable value is the grid + triage queues, which is maybe a third of the panels being built.

## 2. drc atlas — the Graph & Knowledge Explorer

**[threatening] The entire design stands on API endpoints the ground-truth inventory says don't exist, with no designed fallback.** `01-data-inventory.md` states graph-structure export is "**Not wired today** … the single biggest feasibility unknown," and names the fallback: the D5 entity co-occurrence graph, which `00-framework.md` §3 calls "underexploited." Atlas §5 builds on `/datasets/{id}/graph` and `/schema/inventory` (citing "live-verified" research the inventory doc was never updated to reflect), and its drift mitigation is "a clear banner" — i.e., if the unversioned hosted API changes, the surface is a banner. D5 appears nowhere in atlas despite mission §4 calling entity pulse "the seed of the atlas's pre-cognify preview." — *Fix: make the D5 co-occurrence graph atlas's v0 canvas (local, guaranteed data) and layer Cognee snapshots on top as the enhanced mode.*
*(Post-review note from the coordinator: the inventory doc was stale — R8 verified the endpoints live; the doc has since been corrected. The residual valid point is the missing D5 layer and drift posture.)*

**[material] Internal contradiction on layout computation — the doc argues with itself mid-sentence.** §2 P4: "FA2 run once **server-side** per graph snapshot, positions persisted." §5: "FA2 positions (Python-side via graphology in node? no — run FA2 in a **web worker client-side** on first load…)" — an unresolved deliberation left in the spec, and the two sections now disagree. Worse, §3's scenes use "radial layout" pinned to the focus node and the overview uses meta-nodes, so it's unclear any surface actually *displays* the persisted FA2 positions except diff mode; P4's "spatial memory" rationale doesn't match the UX that was designed. — *Fix: pick one computation site and state exactly which view consumes persisted positions.*

**[material] "Bookmarkable scenes" break on every re-cognify, by the design's own admission.** §3: "Scene state (node ids + camera) serialized to the URL hash — every view is a bookmarkable, pasteable artifact." §5 risks: "Node-id instability across cognify runs." Tool graphs are pitched as "long-lived reference artifacts" (§1) that get re-cognified — so every saved link to a long-lived graph dies at the next ingest. — *Fix: serialize scenes as label+type queries (re-resolved on load) rather than raw node ids.*

**[material] Diff mode reintroduces the forbidden hairball.** §1 non-goals: "the hairball is an explicit anti-goal"; §2 P1: "global hairballs fail past ~200 nodes" at a verified 3k nodes / 13k edges. Yet §3 diff mode renders a "union overlay" of two full snapshots — a 3k+ node four-color hairball. The usable artifact ("most rewired entities" list) is the sidecar, not the canvas. — *Fix: make diff scene-scoped (diff the current search's neighborhood) with the rewired list as the global entry point.*

**[minor] Query workbench cost ambiguity.** §2 P5 says hosted credits matter and calls are "explicit user actions," but §4 shows GRAPH_COMPLETION / CHUNKS / SUMMARIES "side by side" for one question — is that one action and three billed searches? — *Fix: tabs fire lazily, one search type per explicit click.*

**[minor] Overview claims to work "before a full export" but needs one.** §4: meta-nodes from `/schema/inventory` counts, *plus* "Louvain communities on the Entity-Entity projection" — Louvain requires the full edge list, so the pre-export overview can only be the type counts. — *Fix: specify a degraded types-only overview until a snapshot exists.*

**Strongest skeptic objection:** the atlas's unique job — browsing graph *structure* — is the weakest-demand job of the three. Answers come from `drc query` (which the workbench merely wraps), and evidence-checking is done better by dossier, anchored to actual claims. Graph explorers demo beautifully and then go unopened; the design half-knows this ("text answers stay primary… the atlas is the verification, orientation, and serendipity layer" — §1), which is a polite way of saying it's the second screen for two jobs other tools own. After week one it survives only if the provenance breadcrumb and gap cards are dramatically better than asking the agent — and both sit on a fuzzy join (§2 P3) the design admits will sometimes degrade to a dead-end "ingested artifact" label.

## 3. drc dossier — the Report & Evidence Studio

**[material] The VENDOR valence heuristic is unimplementable as specified.** §2 P3: VENDOR "when S's URL domain matches **the subject entity's domain**." Per the inventory, entities are bare strings (`entities: list[str]`); nothing maps "Zep" → getzep.com, and nothing identifies which entity is a claim's *subject*. The question-2 promise ("valence at a glance") rests on an unspecified entity→domain resolver plus fuzzy subject detection, stacked on the already-fuzzy USE alignment. — *Fix: derive VENDOR from source metadata only (e.g., category + found_via + a small manual domain map per topic) and drop subject-entity inference.*

**[material] The claim extractor will cry wolf.** §3 audit loop extracts "report sentences containing citations **or ranking language**" and §2 P4 flags "uncited paragraphs" by absence. Ranking-language detection is unspecified NLP, and absence-flagging marks every transition/summary paragraph as a ⚠ gap. A solo user gets one week of false alarms before mentally muting the amber markers — killing question 4 (honest gaps) via alert fatigue. The "parse coverage" stat (§5 risks) measures indexing, not false positives. — *Fix: v1 flags only ranking-section sentences with zero citations, and expand vocabulary only after observing real reports.*

**[material] Grade chips are complexity creep the brief forbids.** §4 introduces a two-axis Admiralty grading system (reliability letter × corroboration number, with legend and tooltip axes) on top of valence badges, chip fill-styles (solid/hollow/gray/dashed-amber), margin notes, and the audit table — five overlapping trust encodings for one reader who already knows her own corpus. This is an intelligence-agency artifact in a personal tool; nobody will re-learn what "B3" means in month three. — *Fix: cut grades; keep valence badge + source count, which the audit table's "weakest first" sort already operationalizes.*

**[material] "Fully useful at level (a)" is asserted, not designed for.** §2 P1 concedes no citation spans exist and stakes v1 on fuzzy-matching *paraphrased* takeaways against long content files, with a ≈ marker when confidence is low and (§5 risks) *no highlight at all* below threshold. If — plausibly — most matches land below threshold, the flagship detail ladder's step 3 opens an un-scrolled full document, which is exactly the pre-dossier experience. The design's fallback for its core interaction is its own absence, yet no expected-hit-rate check or spot-test is planned before the 5.5-day build. — *Fix: run the rapidfuzz match offline against one existing workspace first and gate the build (or pull the `quote`-span pipeline change from "recommendation" to prerequisite) if hit rate is poor.*

**[minor] Half the canvas is dead in the default mode.** §4 marks the far-right band "EVIDENCE PANELS (on click)" — in Read mode, the default posture, a large fraction of the 3840px canvas sits empty until a chip click, which is a weak answer to rubric criterion 5. — *Fix: idle-state the panel with the current viewport's strongest evidence (extending the margin-note logic).*

**Strongest skeptic objection:** a report is read approximately once. Dossier spends ~5.5 build-days on a surface whose read/audit modes serve a 20-minute post-run session, per run. The recurring posture — tool-graph reference — is served by the capability index, which is a filterable rendering of `synthesis/*.md` files the user can already read raw; that one panel, not the studio, is the piece with week-two retention, and it's parked in v1.5.

## Cross-cutting: "the three surfaces form one app" (dossier §3) — do they?

**[material] "Atlas" names two different things.** `drc atlas` is a surface (`/w/<slug>/atlas`); dossier §3/§4 also has an "`Atlas` tab" *inside dossier* that is the capability index, not the graph explorer — the footnote even clarifies the collision ("* Atlas tab appears for tool-graph workspaces"). In a three-surface app, reusing the name of surface 2 for a tab on surface 3 guarantees confusion, especially since that tab *also* contains "open in graph" links to real atlas. — *Fix: rename the dossier tab "Controls" or "Index."*

**[material] `Esc` means something different — and differently destructive — on each surface.** Mission: mark-all-seen + exit inspect (§3); atlas: "clears back to overview," destroying the built-up scene (§3); dossier: collapses evidence panels (§3). Same key: harmless dismiss, irreversible badge-wipe, and scene destruction. For one app driven by muscle memory, this is a daily-papercut generator. — *Fix: one shared keymap doc; Esc = dismiss topmost pane everywhere; scene-clear and mark-seen get their own affordances.*

**[material] Two provenance chains for the same claim can disagree.** Dossier evidence = report → source id → DistilledRecord → fuzzy content match; atlas evidence = node → chunk → TextDocument → fuzzy `ingested.json` join → source id (atlas §2 P3). Both chains are fuzzy at different links, so clicking "evidence" for the same fact on two surfaces of "one app" can produce different sources or different dead-ends, undermining the shared trust story. — *Fix: build one provenance service in the dash chassis that both surfaces consume, with a single confidence/degradation vocabulary.*

**[material] The `viz/` directory becomes unmanaged shared state — the maintenance drag the rubric forbids.** Mission writes `events.jsonl` + (v1.5) `directives.jsonl`; atlas writes `graph-<n>.json` snapshots + persisted positions; dossier caches "invalidated by the same watcher as mission" (dossier §5), coupling its lifecycle to mission's watcher. No design states retention, schema-versioning (except directives), or what happens when three panels' watchers/caches disagree about the same rewritten `sources.jsonl`. — *Fix: one page in the shared chassis spec: viz/ file contract, version fields, and a size/retention rule.*

**[minor] The portfolio busts the personal-tool budget and still misses a moment.** Summed honest estimates: mission ~5-6 days, atlas ~7, dossier ~5.5 — three-plus weeks of solo build against a "weekend-buildable" rubric line (framework §7.3), while M7 (retrospective/cross-run) is explicitly deferred by all three designs, so the rubric's distinctness goal is met by overlapping the exploration moment (atlas evidence vs dossier evidence vs mission entity pulse) rather than covering the lifecycle. — *Fix: pitch a shared-chassis v0 across all three (grid+inspector, D5 scene, chips+hover cards ≈ one week total) and let usage data pick which surface earns its v1.*

**Overall read:** mission is the strongest pitch but its steering/triage story is half-plumbed; atlas is the most fragile (single-API dependency, self-contradicting spec) and most likely to sit unused; dossier has the clearest job but has quietly re-scoped from "verification approximately free" to a heuristics stack whose accuracy nobody has measured. None of the three is unsalvageable; all three need the threatening/material items resolved before the HTML pitch reports get written.
